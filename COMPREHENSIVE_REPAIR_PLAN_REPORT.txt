1. Executive Summary / الملخص التنفيذي
--------------------------------------------------------------------------------
The project demonstrates advanced ambition (multi-service social media integrations, Firebase, GCP stack, localization). However, technical debt, structural drift, insecure secret handling, inconsistent typing, and lack of formal quality controls threaten scalability, maintainability, and compliance.

المشروع متقدم طموح، لكن التراكم التقني وتشتت الهيكل وضعف حوكمة الأكواد وتعريض الرموز السرية للاطلاع العلني وغياب اختبارات منهجية يشكل مخاطر عالية على الاستدامة والإطلاق الإنتاجي الآمن.

Key Critical Themes / المحاور الحرجة:
- Secrets exposed / الرموز الحساسة في العميل
- Mixed frontend/backend concerns / خلط مسؤوليات الواجهة والخلفية
- Absence of structured CI quality gates / غياب بوابات الجودة الآلية
- Type integrity erosion (293 any + 50+ TS warnings) / تآكل سلامة الأنواع
- Logging inconsistency & low observability / تسجيل غير موحد وضعف الرصد
- Rate limiting & API security implemented client-side only / حماية معدلات غير فعّالة
- Duplicate & divergent service files / ملفات مكررة متضاربة

Strategic Goal (90-day Objective): Reach production readiness with auditable, secure, observable, test-backed platform without feature regression.
الهدف الاستراتيجي (90 يوماً): جاهزية إنتاجية آمنة قابلة للرصد والاختبار دون حذف أي ميزة.

4. Classification & Prioritization Model / نموذج التصنيف والأولوية
--------------------------------------------------------------------------------
Severity Levels:
- Critical: Security, data integrity, production-stopping.
- High: Scalability, maintainability, potential data mismatch.
- Medium: Performance, consistency, developer UX.
- Low: Cosmetic, stylistic, redundancy.

Priority Formula (P): P = (Severity Weight × Exploitability × User Impact × Frequency) – (Mitigation Ease Score)
Weights (guideline): Critical=5, High=3, Medium=2, Low=1.

5. Root Cause Analysis / تحليل الجذور
--------------------------------------------------------------------------------
Root Themes:
- Lack of enforced architectural boundaries (frontend vs backend logic).
- Opportunistic feature addition without stabilization iteration.
- Absence of typing discipline & lint gate.
- Inconsistent environment strategy (migration artifacts between CRA and Vite paradigms).
- No formal test pyramid (only ad-hoc mega test file).
- Security assumptions delegated to client layer (invalid for secret-bearing APIs).
- Logging introduced but not backfilled across legacy modules.

================================================================================
6. Remediation Strategy (Phased Plan) / الخطة الإصلاحية المرحلية
--------------------------------------------------------------------------------
PHASE 0 (Preparation – Week 0):
- Freeze new feature merges except security hotfixes.
- Introduce CI skeleton (install → typecheck → lint dry run).
- Inventory secrets & rotate high-risk tokens.

PHASE 1 (Foundation – Weeks 1–2):
- Backend Segregation: Move social media write operations & tokens to secure backend layer (Firebase Functions or Cloud Run façade).
- Environment Unification: Choose Vite or keep CRA temporarily; standardize prefixes.
- Type Blocking Gate: Fail CI on new any or new TS errors.
- Secret Scrub: Remove hard-coded or public-exposed access tokens.

PHASE 2 (Safety Nets – Weeks 3–4):
- Introduce unit tests (auth, messaging, caching, social façade).
- Logger Consolidation: Replace console.* in critical paths.
- Add request correlation IDs and structured log shape.
- Implement SSR-safe / client-safe abstraction for config.

PHASE 3 (Quality Expansion – Weeks 5–7):
- Convert high traffic modules to strict typing (eliminate top 40% any sources).
- Add integration tests (mocked external APIs for TikTok, Instagram, Facebook).
- Add caching validation & stale/refresh strategy.
- Implement backend rate limiting (remove reliance on browser).

PHASE 4 (Optimization – Weeks 8–9):
- Bundle splitting + dynamic imports for heavy services.
- Introduce image optimization pipeline & lazy load strategy.
- Firestore index policy & schema documentation.

PHASE 5 (Hardening & Observability – Weeks 10–11):
- Add metrics: latency, error rate, external API success ratio.
- Add alert thresholds (Pager/Webhook simulation).
- Security review: Firestore rules & auth flows.

PHASE 6 (Stabilization – Week 12):
- Technical debt sweep (remaining any, TODOs).
- Documentation consolidation (single authoritative README + architecture doc).
- Sign-off & readiness assessment.

8. Quality Gates / بوابات الجودة
--------------------------------------------------------------------------------
Gate 1: TypeScript must compile with ≤ existing warning baseline (no net new).
Gate 2: Lint passes with no Critical/High rule violations.
Gate 3: Unit test coverage (lines) ≥ 40% Phase 3, 60% Phase 5.
Gate 4: No secrets in generated build artifacts (scan with regex patterns).
Gate 5: Bundle main chunk ≤ 200 kB (gzipped) by Phase 4.
Gate 6: P95 API façade latency < 400 ms.
Gate 7: Zero unhandled promise rejections in logs.

10. Security & Compliance Guardrails / الأمان والامتثال
--------------------------------------------------------------------------------
- Disallow direct external social API calls from client except read-only if public.
- Enforce HTTPS-only endpoints.
- Add CSP & Referrer-Policy headers (deferred to deployment config).
- Validate all user-generated content (sanitize messaging inputs).
- Apply principle of least privilege on Firestore rules (role-based claims).
