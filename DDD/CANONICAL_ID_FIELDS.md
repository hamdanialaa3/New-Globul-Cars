# Canonical ID Field Naming Convention

## 📋 Overview
This document defines the **canonical naming convention** for ID fields across the Bulgarian Car Marketplace codebase to ensure consistency and prevent confusion.

---

## ✅ Standard Field Names

### 1. **User Identity**
- **Field Name:** `uid`
- **Type:** `string`
- **Usage:** Unique identifier for users (from Firebase Auth)
- **Examples:**
  ```typescript
  // User Profile
  interface BulgarianUser {
    uid: string;  // ✅ Correct
    email: string;
    displayName: string;
  }
  
  // Auth Context
  const { currentUser } = useAuth();
  console.log(currentUser.uid);  // ✅ Correct
  ```

### 2. **Car Ownership**
- **Field Name:** `sellerId`
- **Type:** `string` (required, not optional)
- **Usage:** Owner/seller of a car listing
- **Examples:**
  ```typescript
  // Car Listing
  interface CarListing {
    id?: string;
    sellerId: string;  // ✅ Correct (required)
    sellerEmail: string;
    sellerName: string;
    // ...other fields
  }
  
  // Creating a listing
  const listing: CarListing = {
    sellerId: currentUser.uid,  // ✅ Always equals auth.uid
    sellerEmail: currentUser.email,
    // ...
  };
  ```

### 3. **Social Feed & Posts**
- **Field Name:** `authorId` or `userId` (legacy)
- **Migration Plan:** Migrate from `userId` → `uid` for consistency
- **Examples:**
  ```typescript
  // Current (legacy)
  interface Post {
    id: string;
    userId: string;  // ⚠️ Legacy - will migrate to authorId or uid
    content: string;
  }
  
  // Future (planned)
  interface Post {
    id: string;
    authorId: string;  // ✅ Preferred for social content
    content: string;
  }
  ```

### 4. **Ownership References**
- **Field Name:** `ownerId`
- **Usage:** Generic ownership (use sparingly, prefer specific names like `sellerId`, `authorId`)
- **Examples:**
  ```typescript
  // Generic collection with owner
  interface Resource {
    id: string;
    ownerId: string;  // ⚠️ Use only when sellerId/authorId don't apply
    type: string;
  }
  ```

## 🛡️ Security Rules Integration

### Firestore Rules
```javascript
// Cars Collection
match /cars/{carId} {
  allow create: if isAuthenticated() &&
                   request.resource.data.sellerId is string &&
                   request.resource.data.sellerId == request.auth.uid &&
                   request.resource.data.sellerEmail == request.auth.token.email;
}

// Posts Collection (future)
match /posts/{postId} {
  allow create: if isAuthenticated() &&
                   request.resource.data.authorId == request.auth.uid;
}

// Users Collection
match /users/{userId} {
  allow read: if isAuthenticated();
  allow write: if userId == request.auth.uid;  // uid matches document ID
}
```

## 📚 Quick Reference

| Context | Field Name | Type | Required | Notes |
|---------|-----------|------|----------|-------|
| User identity | `uid` | `string` | Yes | From Firebase Auth |
| Car listing owner | `sellerId` | `string` | Yes | Must equal `auth.uid` |
| Social post author | `authorId` | `string` | Yes | Preferred (migrating from `userId`) |
| Generic ownership | `ownerId` | `string` | Depends | Use sparingly |
| Document ID | `id` | `string` | Optional | Auto-generated by Firestore |

## ❌ Deprecated Patterns (Avoid)

```typescript
// ❌ DON'T: Optional sellerId
interface CarListing {
  sellerId?: string;  // BAD: Should be required
}

// ❌ DON'T: Mixed naming in same context
interface Post {
  userId: string;    // BAD: Inconsistent with uid
  ownerId: string;   // BAD: Redundant
}

// ❌ DON'T: Generic naming for specific contexts
interface CarListing {
  userId: string;    // BAD: Should be sellerId
}
```

---

## ✅ Correct Patterns (Follow)

```typescript
// ✅ DO: Required sellerId for car listings
interface CarListing {
  sellerId: string;  // GOOD: Required and specific
  sellerEmail: string;
}

// ✅ DO: Consistent uid for user identity
const loadUserData = async (uid: string) => {
  const userDoc = await getDoc(doc(db, 'users', uid));
  return userDoc.data() as BulgarianUser;
};

// ✅ DO: Specific names for different contexts
interface Message {
  senderId: string;   // GOOD: Clear role
  receiverId: string; // GOOD: Clear role
  content: string;
}
```

## 🔍 Finding Non-Compliant Code

Use these grep patterns to find inconsistencies:

```bash
# Find optional sellerId (should be required)
grep -r "sellerId\?: string" src/

# Find userId in car contexts (should be sellerId)
grep -r "userId.*car" src/

# Find mixed ID patterns in same file
grep -r "userId.*sellerId\|sellerId.*userId" src/
```
