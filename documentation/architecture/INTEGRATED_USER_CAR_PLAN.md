# 🏗️ خطة التكامل العميق: الهيكلية المنطقية للمستخدم والسيارة
# Deep Integration Plan: User-Car Logical Architecture

تستند هذه الوثيقة إلى المنطق البرمجي العميق لربط الكيان الجذر (المستخدم) بكياناته الفرعية (السيارات) وعملياتها (العرض والتعديل)، لضمان تسلسل هرمي نظيف ومترابط.

This document outlines the architectural logic to bind the Root Entity (User) to its Child Entities (Cars) and their operations (View/Edit), ensuring a clean hierarchical sequence.

---

## 1. الفلسفة المنطقية (The Logical Philosophy)

النظام سيعمل وفق مبدأ **"التسلسل الهرمي للملكية" (Ownership Hierarchy)**.
*   **الأصل (Parent):** المستخدم (رقم 101).
*   **الفرع (Child):** السيارة (رقم 101-1).
*   **الإجراء (Action):** العرض (View) أو التعديل (Edit).

### تمثيل الروابط (URL Logic)
بدلاً من أن تكون السيارات كائنات "عائمة" في النظام، سنقوم بربطها بمسار المالك لتعزيز مفهوم الملكية.

| المفهوم (Concept) | المعرف المنطقي (Logical ID) | مسار الرابط المقترح (Proposed Route) | الوصف (Description) |
| :--- | :--- | :--- | :--- |
| **المستخدم (الجذر)** | `101` | `/profile/101` | الصفحة الرئيسية للمالك. |
| **السيارة (الابن)** | `101-1` | `/profile/101/car/car_id` | رابط السيارة، يظهر بوضوح أنها تابعة للمستخدم 101. |
| **العرض (View)** | `101-1-1` | `/profile/101/car/car_id/details` | تفاصيل السيارة (كما يراها المشتري). |
| **التعديل (Edit)** | `101-1-2` | `/profile/101/car/car_id/edit` | **إعادة فتح خطوات الإضافة الـ 6** للتعديل عليها. |

## 2. توحيد تجربة الإضافة والتعديل (The Unified 6-Steps Logic)

حالياً، عملية "الإضافة" (Add) وعملية "التعديل" (Edit) منفصلتان برمجياً.
لتحقيق منطقك: *"تغيير أي معلومات تمت إضافتها عند الخطوات الـ 6"*، يجب **دمج المسارين**.

### الأسلوب الجديد: `VehicleActionWizard`
بدلاً من الحصول على نموذج تعديل بسيط، سنقوم بتحويل `SellVehicleWizard` ليعمل في وضعين:

1.  **وضع الإنشاء (Create Mode):**
    *   يبدأ ببيانات فارغة.
    *   ينشئ المعرف `101-جديد`.
    *   النتيجة: نشر سيارة جديدة.

2.  **وضع التعديل (Edit Mode) - (المنطق الذي طلبته):**
    *   يستقبل `carId`.
    *   **يقوم بجلب البيانات وتعبئة الخطوات الست (6 Steps) تلقائياً.**
    *   يتيح للمستخدم التنقل بين الخطوات (1. الأساسيات -> 2. الصور -> ... -> 6. الموقع) وتغيير ما يريد.
    *   النتيجة: تحديث (Update) نفس المعرف `101-1`.

## 3. خطة التنفيذ البرمجية (Implementation Roadmap)

### المرحلة 1: إعادة هيكلة التوجيه (Routing Refactoring)
*   [ ] إنشاء مسارات متداخلة (Nested Routes) في `AppRoutes.tsx` او `MainRoutes.tsx`.
*   [ ] تعريف المسار: `/profile/:userId/car/:carId/*` ليكون الحاوية الرئيسية.

### المرحلة 2: ترقية معالج الإضافة (Wizard Upgrade)
*   [ ] تعديل `SellVehicleWizard.tsx` ليقبل `initialData` (البيانات الحالية للسيارة).
*   [ ] إضافة منطق `isEditMode` داخل الـ Wizard.
*   [ ] عند فتح رابط التعديل (`101-1-2`)، يتم تحميل الـ Wizard بدلاً من صفحة التعديل البسيطة الحالية.

### المرحلة 3: أمن البيانات (Data Integrity)
*   [ ] التأكد من أن المستخدم `101` هو الوحيد الذي يمكنه دخول المسار `101-1-2`.
*   [ ] استخدام "الحذف الناعم" (Soft Delete) الذي قمنا ببنائه كخيار داخل هذه اللوحة.
