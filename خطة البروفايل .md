# خطة هيكلة البروفايل الاحترافية وربطه بنظام إضافة السيارة

هدف هذه الخطة: تحويل البروفايل إلى مركز التحكم الأساسي لكل أنواع البائعين (شخصي، تاجر، شركة)، بحيث تُدار تجربة إضافة السيارة كاملةً من البروفايل وفق نوع الحساب، مع التزام صارم بقيود المشروع والبيئة البلغارية.

## 1) أنواع البروفايل وخصائص كل نوع

- شخصي (Private Person)
	- الألوان والهوية: تبقى كما هي (الهوية الحالية البرتقالية).
	- الصلاحيات: عدد إعلانات محدود، بدون أقسام تجارية، تواصل مباشر بسيط.
	- المتطلبات: تحقق بسيط من الهوية ووسيلة اتصال.

- تاجر/معرض سيارات (Dealer)
	- الألوان والهوية: أخضر بالكامل ضمن صفحات البروفايل المتأثرة.
	- الصلاحيات: إدارة مخزون سيارات، خيارات تمويل، ضمان، صفحة معرض عامة، تقارير أداء.
	- المتطلبات: بيانات نشاط تجاري بلغاري، تسجيل في الضرائب إن لزم، عنوان تجاري.

- شركة (Company)
	- الألوان والهوية: أزرق، مظهر احترافي/بزنس، الصورة الشخصية مربعة في الوسط مع تأثير LED أزرق محيط رفيع.
	- الصلاحيات: إدارة أسطول (Fleet)، شروط خاصة، عقود شركات، صلاحيات فرق العمل.
	- المتطلبات: رقم الشركة البلغاري (EIK/BULSTAT)، عنوان، شخص مفوض.

ملاحظة: الهوية البصرية تُطبق فقط ضمن مساحات البروفايل ولوحاته، مع الحفاظ على ثبات تصميم بقية الموقع.

## 2) بنية التوجيه والربط مع صفحات المشروع

- البروفايل: `/profile` (مطلوب تسجيل الدخول)
	- تبويب اختيار وضع الحساب (Profile Mode Switcher) في أعلى الصفحة: شخصي | تاجر | شركة.
	- عند التبديل، تظهر نافذة تأكيد تسبقها صفحة شروط موجزة خاصة بنوع الحساب.

- نظام البيع الحالي يندمج مع البروفايل حسب النوع:
	- القسم 1 (بيانات أساسية): `/sell/inserat/:vehicleType/fahrzeugdaten/antrieb-und-umwelt`
	- القسم 2 (التجهيزات): `/sell/inserat/:vehicleType/equipment`
	- القسم 3 (السعر والاتصال/الصور): `/sell/inserat/:vehicleType/contact` + مسارات الصور والسعر القائمة
	- تظل الروابط العامة كما هي في "صفحات المشروع كافة .md" لضمان التوافق، لكن نقطة الانطلاق تصبح من البروفايل مع تهيئة مسبقة حسب نوع الحساب.

## 3) تجربة المستخدم (UX) حسب نوع البروفايل

- شخصي:
	- خطوتان مبسّطتان + ملخص نهائي.
	- تواصل مباشر، إخفاء البيانات الحساسة حتى تفاعل جاد.

- تاجر:
	- معالج إضافة سيارة يحتوي على: بيانات أساسية → تجهيزات → صور → سعر → مزايا تاجر (تمويل/ضمان) → ملخص.
	- لوحة مخزون، استيراد/تصدير بيانات سيارات (CSV)، صفحة معرض عامة.

- شركة:
	- معالج يراعي إدارة أسطول: دفعات سيارات، سياسات تسعير، شروط عقود، جهات اتصال متعددة.
	- إدارة صلاحيات الفريق داخل البروفايل، مراجعة داخلية قبل النشر.

## 4) المتطلبات القانونية البلغارية (تلخيص توجيهي للرسائل الشرطية)

- البيانات التجارية: رقم الشركة EIK/BULSTAT، العنوان المسجل، الشخص المفوض.
- الضرائب: التسجيل في NRA عند انطباق شروط النشاط الاقتصادي المنتظم.
- حماية المستهلك: شفافية السعر والحالة والضمان عند البيع كتاجر/شركة.
- حماية البيانات: التوافق مع GDPR، عرض سياسة الخصوصية والموافقة.

ملاحظة: تُعرض هذه الشروط في شاشة التأكيد قبل تفعيل وضع "تاجر" أو "شركة"، ويمكن تحديث النصوص قانونيًا لاحقًا.

## 5) نموذج البيانات والقيود

- UserProfile
	- id, email, displayName, phone
	- profileType: 'private' | 'dealer' | 'company'
	- businessInfo?: { legalName, eik, address, taxRegistered: boolean }
	- companyInfo?: { fleetSize, contractOptions }
	- themeToken (colors per type):
		- private: { primary: '#ff8f10' }
		- dealer: { primary: '#16a34a' }
		- company: { primary: '#1d4ed8' }

- Listing (Car)
	- ownerId, ownerType (derives from profileType at creation time)
	- vehicleCore, equipment[], media[], price, contact
	- complianceFlags: { warrantyOffered?, financing?, commercialSale? }

قيود:
	- حجم الملفات والصور وفق `storage.rules` الحالية.
	- عدم تجاوز 300 سطر للملف الواحد: تقسيم React/Services إلى وحدات صغيرة مع تعليقات مختصرة واضحة.
	- لا تكرار: مكونات مشتركة (Layout, Forms, Validation) يعاد استخدامها.

## 6) الصلاحيات والتدقيق

- شخصي: إنشاء/تعديل/حذف إعلاناته فقط، سقف إعلانات شهري.
- تاجر: سقف أعلى، أدوات مخزون، نشر سريع.
- شركة: صلاحيات فرق (roles): Owner, Manager, Editor.
- تدقيق (Audit): حفظ أحداث التغيير الأساسية في Firestore.

## 7) التوطين والعملات

- اللغات: بلغاري/إنكليزي. النصوص عبر i18n.
- العملة: يورو. تنسيق الأسعار ثابت، مع إمكانية التحويل العرضي لاحقًا.

## 8) واجهة التبديل والهوية البصرية داخل البروفايل

- شريط علوي فيه 3 أزرار (Private | Dealer | Company).
- عند اختيار Dealer: تطبيق سمة خضراء على أقسام البروفايل ذات الصلة. الصورة دائرية في الوسط بحجم 180%.
- عند اختيار Company: سمة زرقاء، صورة مربعة في الوسط مع تأثير LED أزرق رفيع حول الصورة وحواف البروفايل.
- يمنع استخدام رموز تعبيرية نصية؛ الأيقونات من svgrepo فقط.

## 9) الدمج مع مسار البيع الحالي

- يبدأ المعالج دائمًا من البروفايل ليقرأ نوع الحساب.
- يُمرَّر نوع الحساب إلى صفحات البيع عبر Query/Context لتكييف الحقول (مثال: إظهار تمويل/ضمان للتاجر فقط).
- يحافظ على المسارات الحالية المذكورة في "صفحات المشروع كافة .md" لضمان الروابط العامة.

## 10) الأمان والخصوصية والامتثال

- اتباع قواعد الحذف (Delete User Data) الموثقة.
- قواعد Firestore/Storage المضبوطة (مجلدات public/users/cars).
- تجنّب تسريب البيانات الشخصية لغير المخوّلين.

## 11) خطة التنفيذ على مراحل (Milestones)

- المرحلة 1: تمهيد البروفايل
	- إضافة `profileType` للملف الشخصي وواجهة تبديل الوضع مع شاشات الشروط.
	- تطبيق السمات البصرية داخل البروفايل فقط.

- المرحلة 2: ربط مسار البيع
	- تمرير نوع الحساب إلى صفحات البيع وتكييف الحقول والملخص.
	- حفظ المسودات وربطها بالحساب.

- المرحلة 3: ميزات متقدمة حسب النوع
	- للتاجر: إدارة مخزون، تمويل/ضمان.
	- للشركة: إدارة أسطول وصلاحيات فريق.

- المرحلة 4: التدقيق والتقارير
	- سجلات أحداث، ملخصات أداء، تصدير CSV للتاجر والشركة.

## 12) حدود الأداء والجودة

- تقسيم الملفات الكبيرة >300 سطر إلى وحدات.
- تعليقات مختصرة، دوال واضحة، لا تكرار.
- اختبار أساسي قبل النشر، والالتزام بأن ما يُطبق صالح فعليًا للإطلاق العام.
