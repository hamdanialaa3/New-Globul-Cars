**الغاية:** تطبيق موبايل متماثل مع الواجهة الأمامية للمشروع الأصلي ومرتبط بنفس البنية الخلفية

## الجزء الأول: نص التخاطب للنموذج الذكي في Visual Studio Code

انسخ النص التالي والصقه في النموذج الذكي (Cursor / Copilot / أي AI) داخل مشروع Koli One الأصلي (مجلد المشروع الويب)، ثم أرسل لي رد النموذج كما هو لأساعده في بناء تطبيق الموبايل عليه.

### نص التخاطب (انسخ من هنا)

```
أحتاج منك وصفاً مؤسسياً لمشروع Koli One لاستخدامه في بناء تطبيق موبايل مماثل ومرتبط به. اقرأ الدستور (CONSTITUTION.md) والوثائق الأساسية ثم أجب بالتفصيل على النقاط التالية بشكل منظم ومرقّم:

1) أساسيات المشروع
   - اسم المشروع، الموقع (بلغاريا)، اللغات (بلغاري، إنجليزي)، العملة (يورو).
   - الغرض من التطبيق (سوق سيارات: بيع/شراء/إعلانات).

2) نظام الروابط والمعرفات (من الدستور)
   - نمط رابط البروفايل: /profile/:numericId و /profile/view/:numericId والفرق بينهما، ومتى يُحوَّل المستخدم من واحد إلى الآخر.
   - نمط رابط الإعلان: /car/:sellerNumericId/:carNumericId و /car/:sellerNumericId/:carNumericId/edit.
   - نمط الرسائل: /messages/:senderId/:recipientId.
   - قاعدة: ممنوع استخدام Firebase UID في الروابط العامة؛ فقط Numeric ID.

3) المصادقة والمستخدمون
   - طرق تسجيل الدخول (Google وغيرها، دخول كضيف).
   - كيف يُعطى المستخدم numericId عند أول تسجيل.
   - الفرق بين صفحة المستخدم لنفسه (/profile/:id) وصفحة عرض بروفايل غيره (/profile/view/:id).

4) الوظائف الأساسية (Features) المطلوب تكرارها في الموبايل
   - البحث: بسيط (/cars)، متقدم (/advanced-search)، ذكي (mode=smart).
   - عرض الإعلانات: التصنيفات (passenger_cars, suvs, vans, motorcycles, trucks, buses)، الحقول الرئيسية في CarListing (make, model, year, price, mileage, fuelType, transmission, city, images, status, إلخ).
   - البروفايل: إعلانات المستخدم، المفضلة، الإعدادات، الاشتراكات.
   - الرسائل: بدء المحادثة من صفحة سيارة أو بروفايل، الإشارة للسيارة في سياق المحادثة.
   - المفضلة (Favorites)، البحث المحفوظ (Saved Searches).

5) الخدمات والبنية الخلفية
   - Firebase: Firestore، Auth، Storage، Cloud Functions (أسماء الدوال الأساسية التي يتعامل معها الويب: مثلاً الإشعارات، الرسائل، مزامنة Algolia، إلخ).
   - Algolia: هل مُفعّل؟ اسم الفهرس (مثلاً cars_bg)، الاستخدام في البحث المتقدم.
   - الخدمات الأمامية الأساسية: smartSearchService، UnifiedSearchService، algoliaSearchService، وسائل جلب السيارات من المجموعات المتعددة.

6) واجهة المستخدم والتصميم
   - الألوان: Primary (مثلاً #8b5cf6)، التدرج، الخلفية الداكنة، ألوان النص والحدود.
   - المكونات الرئيسية: SmartAutocomplete، CarCard/Compact، أقسام البحث المتقدم (Basic, Technical, Exterior, Interior, Location، إلخ).
   - قاعدة: منع الرموز التعبيرية (emojis) في الواجهة حسب دستور المشروع.

7) التوطين (i18n)
   - الملفات: locales/bg و locales/en، المفاتيح المستخدمة في advancedSearch و common و carDetails.
   - أي قيم ثابتة خاصة ببلغاريا (مدن، فئات وقود، ناقل حركة، إلخ) يجب أن تكون قابلة للترجمة أو موحّدة.

8) الاشتراكات والدفع
   - الخطط: Free، Dealer، Company مع حدود الإعلانات والسعر (يورو).
   - مصدر الحقيقة: subscription-plans.ts، والتدفق مع Stripe أو التحويل البنكي إذا كان مُطبّقاً.
   - ما يهم للموبايل: عرض الخطط، حدود الإعلانات، الترقية.

9) قواعد إضافية من الدستور
   - حد 300 سطر للملف البرمجي؛ عدم التكرار؛ رمي الملفات للحذف إلى مجلد DDD وليس الحذف المباشر.
   - أسماء المتغيرات والوظائف بالإنجليزي؛ التعليقات مفيدة.

10) أي تفاصيل تقنية إضافية تهم تطبيق موبايل
    - Firestore rules (قراءة/كتابة الإعلانات، البروفايل، الرسائل).
    - بنية مستند Car في Firestore (أهم الحقول للعرض والبحث).
    - كيف يتم ربط "رسالة" بسيارة محددة في المحادثة.

أجب بصيغة منظمة (عناوين، قوائم، أمثلة أكواد إن لزم) بحيث يمكن لشخص آخر أن يبني تطبيق موبايل يطابق الوظائف والروابط والتصميم بدون الحاجة لقراءة كل الكود.
```

## الجزء الثاني: خطة متكاملة لتطبيق الموبايل

### 1. اختيار التقنية

| الخيار | الإيجابيات | السلبيات | التوصية |
|--------|------------|----------|---------|
| **React Native + Expo** | نفس بيئة React، Firebase SDK متوافق، إمكانية مشاركة الأنماط والمنطق | يحتاج تكيّف مكوّنات الويب | **موصى به** لأولوية الشبه بالواجهة والتكامل مع Firebase |
| Flutter | أداء جيد، واجهة موحدة | بيئة مختلفة، إعادة كتابة الخدمات | بديل قوي |
| Capacitor (لف الويب) | أسرع انطلاقاً، كود واحد | تجربة أقل "native"، حجم أعلى | للموجة الأولى السريعة فقط |

**القرار المقترح:** React Native + Expo مع مشاركة:
- Firebase (نفس المشروع: `fire-new-globul`)
- التكوين (بيئة، قيم ثابتة)
- أنواع TypeScript (CarListing, SearchData, User, إلخ) عبر حزمة مشتركة أو نسخ منضبطة

### 4. الربط بالمشروع الأصلي

| الجانب | أين في الويب | كيف في الموبايل |
|--------|--------------|------------------|
| Firebase | fire-new-globul | نفس `firebaseConfig` و`googleServices` |
| الروابط | React Router | Expo Router أو React Navigation بنفس الأنماط |
| Numeric ID | numeric-id-assignment, numeric-id-lookup | استدعاء نفس Firestore أو Cloud Functions إن مُعرّفة كـ HTTP |
| البحث | smartSearchService, UnifiedSearchService, algoliaSearchService | نفس الخدمات أو نسخ RN مع نفس المداخل والمخارج |
| الرسائل | Realtime DB / Firestore + onNewMessage, notifyOnNewMessage | نفس المجموعات والبنية؛ FCM للموبايل |
| الاشتراكات | subscription-plans.ts, Stripe/تحويل | نفس الخطط والحدود؛ واجهة دفع مناسبة للموبايل |
| i18n | locales/bg, locales/en | مفاتيح مماثلة؛ قد ننسخ الترجمة أو نستوردها |

### 5. ما يُنسخ أو يُستورد من المشروع الأصلي

- **يُنسخ ويُكيّف:**
  - واجهات TypeScript: `CarListing`, `SearchData`, `User`, وقطع من `algolia-record-template`.
  - الثوابت: أسماء المجموعات، حقل `status: 'active'`، إعدادات البحث.
  - مفاتيح i18n (أو الملفات كلها) من `locales/bg` و `locales/en`.
- **يُشارك عبر Firebase فقط (بدون كود مشترك في البداية):**
  - Firestore، Auth، Storage، Cloud Functions.
- **لاحقاً في مونوريبو:**
  - `packages/shared`: أنواع، ثوابت، تكوين، وأي منطق بحث يمكن استخراجه دون ربط بـ DOM.

### 6. نصائح لاختصار الوقت

1. **أول ما يبني الشاشات:** الرئيسية، البحث البسيط، قائمة سيارات، تفاصيل سيارة، بروفايل.
2. **أعد استخدام Firebase Rules و Cloud Functions كما هي** دون تغيير يخص الويب فقط.
3. **Algolia:** إذا كان مُفعّلاً في الويب، استخدم نفس `APP_ID`, `SEARCH_KEY`, `INDEX` في الموبايل لتسريع البحث المتقدم.
4. **الدستور:** اطبع قواعد الروابط وNumeric ID وعلقها في المشروع؛ كل شاشة تمس البروفايل أو الروابط يجب أن تطابقها.

## الجزء الثالث: التوجيه من الصفر

### الخطوة 0: ما تحتاجه الآن
- [ ] رد النموذج الذكي من المشروع الأصلي (نص التخاطب في الجزء الأول).
- [ ] هذا الملف `MOBILE_APP_MASTER_PLAN.md`.
- [ ] الوصول إلى `CONSTITUTION.md` ومجلد `web_project_doc` للمرجع.

### الخطوة 1: إنشاء رد أساسيات المشروع
1. افتح مجلد المشروع الأصلي (الويب) في VS Code/Cursor.
2. انسخ "نص التخاطب" من الجزء الأول.
3. الصقه في النموذج الذكي وارسله.
4. احفظ الرد في ملف، مثلاً: `PROJECT_FUNDAMENTALS_FROM_AI.md` داخل `New G Cars APP`.

### الخطوة 2: تهيئة مشروع الموبايل
- إذا كان `New G Cars APP` فارغاً أو للموبايل فقط:
  ```bash
  cd "C:\Users\hamda\Desktop\New G Cars APP"
  npx create-expo-app@latest . --template tabs
  ```
- إذا كان فيه محتوى وثائق فقط، أنشئ مجلداً فرعياً، مثلاً `mobile`، وتهيئ فيه:
  ```bash
  mkdir mobile
  cd mobile
  npx create-expo-app@latest . --template tabs
  ```

### الخطوة 3: ربط Firebase
- انسخ من وحدة Firebase في المشروع الأصلي:
  - `apiKey`, `authDomain`, `projectId`, `storageBucket`, `messagingSenderId`, `appId`.
- في Expo: `npx expo install @react-native-firebase/app @react-native-firebase/firestore @react-native-firebase/auth @react-native-firebase/storage` (أو استخدم Firebase JS SDK إذا فضّلت عدم استخدام الواجهة الأصلية في البداية).
- ضع التكوين في متغيرات بيئة أو ملف `config/firebase.ts` (لا ترفع المفاتيح السرية إلى Git).

### الخطوة 4: أول شاشات
1. **الرئيسية:** شريط بحث بسيط + روابط: بحث متقدم، بروفايل، رسائل.
2. **قائمة سيارات:** استدعاء بسيط لـ Firestore (مثلاً `passenger_cars` مع `isActive==true` و `limit(20)`) وعرض في قائمة.
3. **تفاصيل سيارة:** مسار `/car/:sellerNumericId/:carNumericId` وعرض الحقول الأساسية من `CarListing`.
4. **بروفايل:** مساران `/profile/:id` و `/profile/view/:id` مع تطبيق قواعد التوجيه من الدستور.

### الخطوة 5: عندما تريد الدمج أو المونوريبو
- **دمج في مشروع واحد (مونوريبو):**
  - أنشئ `Koli-One-Platform` مع `apps/web` و `apps/mobile` و `packages/shared`.
  - انقل كود الويب إلى `apps/web` وكود الموبايل إلى `apps/mobile`.
  - انقل الأنواع والثوابت والتكوين إلى `packages/shared` واطرق الاستيراد.
- **الإبقاء مستقلين:**
  - أبقِ `New G Cars APP` وموقع المشروع الأصلي كمجلدين منفصلين، مع مشاركة Firebase فقط وتحديث `shared` يدوياً أو عبر نسخ عند تغيير الأنواع/الثوابت.

### الخطوة 6: النشر
- **ويب:** لا يتغير؛ يبقى في `fire-new-globul` و koli.one.
- **موبايل:** EAS Build + إرسال إلى App Store / Google Play؛ يمكنك استخدام نفس دومين روابط عميق (koli.one) للربط مع الويب عند الحاجة.
