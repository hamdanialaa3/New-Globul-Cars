Findings
Data sourcing: /users pulls directly from the users collection via client-side Firestore queries without going through UserRepository, bypassing shared validation/transforms and causing type drift (e.g. stats/verification shapes diverge from the canonical BulgarianUser contract defined in createOrUpdateBulgarianProfile which omits those sub-objects). This also means no central place to enforce pagination, auth, or privacy of email addresses. orderBy('createdAt','desc') will throw if any user document is missing that field; legacy entries synced via syncAuthToFirestore only populate createdAt when the owner runs it manually, so the page relies on an optional, unscheduled admin action and still exposes raw emails.
Lack of localization: Almost every UI string inside the bubble/list/online components is hard-coded in English, ignoring useLanguage(). Examples include button labels (“Bubbles”, “Grid”, “List”), presence text (“Active Now”, “(n online)”), quick actions, and hover cards, so language switching yields a mixed-language experience.
Navigation and UX: Cards push users to /profile/:id with window.location.href, forcing a full reload instead of SPA navigation. The page also uses blocking alert() for unauthenticated follow/message attempts and to signal the unfinished messaging feature, producing a jarring user flow.
Derived data: Quick stats re-count filteredUsers, so totals only reflect the currently loaded batch rather than the directory’s population. The trust score card averages verification.trustScore, but in the shared type that score belongs under stats; the mismatch yields zeros for new profiles. Likewise, the filters rely on accountType that isn’t guaranteed by the canonical model, so large portions of the dataset won’t respond to the filter.
Presence display: “Online now” is determined by whatever isOnline flag happened to be stored when the static query ran; no realtime listener or Cloud Function updates keeps this current, so users rapidly go stale while still being displayed as online.
Error handling: Network failures just log to the console and leave the page blank because there’s no retry or error state. handleFollow doesn’t wrap the async call, so Firestore errors fall through silently, while the loading spinner and “load more” button have no guard for repeated clicks during the same fetch.
Code duplication: UserProfile is declared twice (one inline, one in types.ts), and mini translation tables repeat in component scope, increasing divergence risk. The dedicated module isn’t even imported.
Styling/cleanup debt: Debug console.log calls remain in production paths, and the styled component bundle for a single page is now ~600 lines with multiple duplicated utility styles, making maintenance difficult.
Backend follow workflow: followService increments counters optimistically but doesn’t verify profile existence or wrap messaging in transactions with host rules; in failure scenarios counts can be off. sendFollowNotification writes to a top-level notifications collection with no partitioning or TTL, so the directory adds load to an unindexed collection yet the UI never reads those notifications.
Security/privacy: The directory exposes email and companyName for every profile to any visitor without considering consent or plan limits, and it fetches up to 1000 follow relations for the current user regardless of plan tier or privacy settings.
Firestore indexes: The query stack (orderBy('createdAt','desc') plus future filters on accountType or location) requires composite indexes that are not declared. Adding a where (region/account type) after initial load will immediately fail unless those indexes are built.
Extensibility gaps: There’s no integration with Algolia or server-side search, so filtering large datasets will hit client limits quickly. Missing pieces include server pagination, analytics logging, and rate limiting on follow actions.
Suggested Next Steps
Route all user reads through UserRepository (or at least a dedicated service hook) that respects the canonical BulgarianUser type, performs localization-friendly mapping, and hides sensitive fields based on viewer role. Add server-controlled pagination (cloud function or callable) to stop exposing raw Firestore queries to the client.
Internationalize every user-facing string in UsersDirectoryPage, OnlineUsersRow, and UserBubble, wiring them to the shared translation system.
Replace window.location.href with useNavigate, swap alert() for contextual toasts/modals, and surface proper error states for failed loads or follow actions.
Store real-time presence in a separate collection or Realtime DB presence service, listen with onSnapshot, and cap the UI display to fresh status updates.
Consolidate the duplicated UserProfile interface by importing from types.ts, and move the inline translation object into the global translation files.
Remove console logs and extract shared styled components into reusable modules to keep the page maintainable.
Provide Firestore index definitions for the combined filters, enforce security rules around email exposure, and schedule the Auth→Firestore sync so createdAt is always present.
Add guardrails in followService: wrap operations in a single transaction (instead of two separate updates), ensure notifications are written to a sharded collection, and add plan-based limits to the getFollowing size.
